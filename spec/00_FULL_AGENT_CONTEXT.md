# Техническое задание
# Продукт: «Дашборд рассадки сотрудников»

## 1. Цель проекта

Создать локальное интерактивное frontend-приложение для планирования переезда сотрудников из старых офисов в новые офисы.

Приложение должно позволять:

- создавать несколько сценариев рассадки;
- добавлять старые и новые офисы;
- задавать зоны внутри новых офисов;
- добавлять команды и сотрудников;
- распределять команды, части команд и отдельных сотрудников по офисам и зонам;
- использовать drag-and-drop;
- видеть заполняемость офисов и зон;
- видеть переполнение в числовом виде, например: `Переполнение: 17 мест`;
- видеть ошибки, предупреждения и информационные статусы;
- сравнивать сценарии между собой;
- импортировать данные из Excel `.xlsx`;
- экспортировать проект и отчеты.

Приложение должно быть локальным, offline, без backend.

## 5. Офисы

Типы офисов:

1. Старый офис.
2. Новый офис.
3. Системный офис `Удаленка`.

### 5.1. Старый офис

Старый офис — справочник текущей локации. Он не имеет зон и не участвует в расчете заполняемости новых офисов.

Поля:

- название;
- площадь;
- текущая вместимость;
- комментарий.

### 5.2. Новый офис

Новый офис используется для планирования рассадки.

Поля:

- название;
- площадь;
- зоны и их вместимость;
- комментарий;
- признак черновика.

Адрес, город, этаж и бизнес-центр не нужны.

### 5.3. Зоны нового офиса

Каждый новый офис должен иметь хотя бы одну зону.

Если пользователь не создал зоны, система автоматически создает зону:

```text
Опенспейс
```

Стандартные зоны:

- Кабинеты;
- Опенспейс;
- VIP-кабинеты.

Также можно добавлять кастомные зоны.

Для кабинетов пользователь вводит только общее количество мест.

Для VIP-кабинетов правило:

```text
1 VIP-кабинет = 1 место
```

### 5.4. Офис-черновик

Офис может иметь нулевую вместимость только как черновик.

Если черновой офис используется в сценарии, показывать предупреждение.

### 5.5. Системный офис «Удаленка»

Удаленка реализуется как отдельный системный офис назначения.

Особенности:

- не имеет лимита вместимости;
- не считается физическим офисом;
- не участвует в расчете вместимости новых офисов;
- сотрудники в удаленке считаются размещенными;
- сотрудники в удаленке уменьшают KPI `Требуется посадочных мест`.

## 6. Команды

Поля команды:

- название команды;
- количество сотрудников;
- текущий офис;
- признак VIP / руководство;
- можно делить: да / нет;
- комментарий.

Команда может быть разделена между несколькими офисами и зонами, если включена настройка `Можно делить`.

Если команда не может делиться, но пользователь пытается разделить ее, показывать предупреждение.

VIP-логика:

- VIP-зона доступна всем;
- если обычная команда размещена в VIP-зоне, показывать предупреждение;
- если VIP-команда размещена не в VIP-зоне, показывать предупреждение.

## 7. Сотрудники

Поля сотрудника:

- ФИО;
- должность;
- команда;
- текущий офис;
- признак VIP / руководство;
- формат работы;
- комментарий.

Форматы работы:

- офис;
- гибрид;
- удаленка.

Гибридный формат учитывается только справочно и не влияет на расчеты мест.

FTE не используется. Все сотрудники считаются целыми единицами.

### 7.1. Два режима работы

Приложение должно поддерживать:

1. Агрегированный режим — работа с командами и численностью.
2. Персональный режим — работа со списком сотрудников.

Приложение должно работать даже без импорта списка сотрудников.

### 7.2. Распределение сотрудников

Основная логика — распределение команд.

Но пользователь может раскрыть команду и распределить конкретных сотрудников.

Если команда разделена между офисами, система должна показывать, какие сотрудники куда попали.

### 7.3. Поиск и фильтры

Во вкладке `Сотрудники` должны быть:

- поиск по ФИО;
- фильтр по команде;
- фильтр по текущему офису;
- фильтр по новому офису;
- фильтр по VIP;
- фильтр по формату работы;
- фильтр по статусу размещения.

## 8. Размещения

Размещение — запись о том, куда направлена команда, часть команды или конкретный сотрудник.

Типы размещений:

- командное размещение;
- индивидуальное размещение сотрудника.

Поля размещения:

- тип: команда / сотрудник;
- ID команды или ID сотрудника;
- количество сотрудников;
- офис назначения;
- зона назначения;
- комментарий.

Для индивидуального сотрудника количество всегда равно `1`.

Пользователь должен уметь:

- создать размещение;
- изменить размещение;
- удалить размещение;
- переместить размещение drag-and-drop;
- редактировать размещение через таблицу;
- редактировать размещение через правую боковую панель;
- быстро изменить через popup.

## 9. Drag-and-drop

Drag-and-drop — must-have. Его нельзя упрощать.

Где используется:

- на главном дашборде;
- во вкладке `Распределение`.

Что можно перетаскивать:

- команду;
- часть команды / allocation-блок;
- конкретного сотрудника.

### 9.1. Перетаскивание команды

При перетаскивании команды в офис или зону:

1. Открывается popup.
2. Пользователь указывает количество сотрудников.
3. Если команда уже частично размещена, popup предлагает остаток.
4. После подтверждения создается размещение.

### 9.2. Перетаскивание сотрудника

При перетаскивании сотрудника:

- сотрудник сразу назначается в выбранный офис / зону;
- если не-VIP сотрудник попал в VIP-зону, показывать предупреждение;
- если VIP-сотрудник попал не в VIP-зону, показывать предупреждение.

## 11. Расчеты

### 11.1. Вместимость офиса

```text
Вместимость офиса = сумма вместимости всех зон офиса
```

### 11.2. Занятость офиса

```text
Занято в офисе = сумма сотрудников, размещенных в зонах этого офиса
```

### 11.3. Свободные места

```text
Свободно = вместимость - занято
```

Если значение меньше 0, показывать переполнение.

### 11.4. Переполнение офиса

```text
Переполнение офиса = занято в офисе - вместимость офиса
```

Показывать только если значение больше 0.

Формат:

```text
Переполнение: 17 мест
```

### 11.5. Заполненность офиса

```text
Заполненность офиса (%) = занято / вместимость × 100
```

### 11.6. Занятость зоны

```text
Занято в зоне = сумма сотрудников в зоне
```

### 11.7. Переполнение зоны

```text
Переполнение зоны = занято в зоне - вместимость зоны
```

Формат:

```text
Переполнение: N мест
```

### 11.8. Всего сотрудников

```text
Всего сотрудников = все сотрудники сценария
```

Если список сотрудников не используется, брать сумму численности команд.

### 11.9. Требуется посадочных мест

```text
Требуется посадочных мест = все сотрудники - сотрудники в удаленке
```

### 11.10. Свободный резерв мест

```text
Свободный резерв = вместимость новых офисов - сотрудники, размещенные в офисных зонах
```

Нужна общая сумма и детализация по зонам.

### 11.11. Общее переполнение

Показывать отдельно:

- переполнение офисов;
- переполнение зон.

Не объединять в одно число.

## 12. Пороги и цвета

Пороговые значения должны быть редактируемыми.

Дефолт:

- зеленый — до 85%;
- желтый — от 85% до 100%;
- красный — выше 100%;
- серый — нет данных;
- синий — информационный статус.

Эти настройки сохранять в JSON-проекте.

## 13. Статусы и валидация

Уровни сообщений:

- ошибка;
- предупреждение;
- информация.

### 13.1. Ошибки

- команда распределена больше своей численности;
- сотрудник размещен дважды.

### 13.2. Предупреждения

- офис переполнен;
- зона переполнена;
- сотрудник не размещен;
- команда не полностью распределена;
- обычная команда / сотрудник размещены в VIP-зоне;
- VIP-команда / VIP-сотрудник размещены не в VIP-зоне;
- офис-черновик с нулевой вместимостью используется в сценарии.

### 13.3. Информация

- сотрудник находится на удаленке;
- есть сотрудники на удаленке.

Команда частично на удаленке — нормальное состояние.

## Расчеты

Вместимость офиса = сумма вместимости зон.

Занято в офисе = сумма сотрудников в зонах офиса.

Свободно = вместимость - занято.

Переполнение = занято - вместимость, если значение больше 0.

Переполнение обязательно показывать числом:

```text

```

Заполненность (%) = занято / вместимость × 100.

Для зоны считать аналогично.

- Всего сотрудников;
- Требуется посадочных мест;

- Распределено в офисы;
- На удаленке;
- Не размещено;

- Переполнение офисов;
- Переполнение зон;
- Количество предупреждений;
- Количество ошибок.

- желтый 85–100%;
- красный выше 100%;
- серый нет данных;

- Можно создать индивидуальное размещение сотрудника.
- Можно редактировать размещения вручную.
- Можно удалить размещение.
- Сотрудник в индивидуальном размещении занимает 1 место.

## 7. Drag-and-drop

- Можно перетащить команду.
- Можно перетащить часть команды / allocation-блок.
- Можно перетащить сотрудника.
- При перетаскивании команды открывается popup с количеством.
- Для частично размещенной команды предлагается остаток.
- VIP-конфликты показываются как предупреждения.

## 8. Расчеты

- Вместимость офиса = сумма зон.
- Занятость офиса считается по размещениям.
- Свободные места считаются правильно.
- Переполнение считается правильно.
- Переполнение отображается как `Переполнение: N мест`.
- Заполненность считается в процентах.
- Отдельно считается переполнение офисов и переполнение зон.

## 9. Дашборд

- Есть KPI-блок.
- Есть общий прогресс переезда.
- Есть карточки офисов.
- Есть progress bars.
- Есть блок `Проблемные офисы`.
- Есть панель предупреждений.

## 10. Валидация

Показываются ошибки:

- команда распределена больше численности;
- сотрудник размещен дважды.

Показываются предупреждения:

- офис переполнен;
- зона переполнена;
- сотрудник не размещен;
- команда не полностью распределена;
- обычный сотрудник / команда в VIP-зоне;
- VIP-сотрудник / команда не в VIP-зоне;
- черновой офис с нулевой вместимостью используется.

## 11. Импорт / экспорт

- JSON export работает.
- JSON import работает.
- Excel import `.xlsx` работает или реализован базово.
- Excel template можно скачать.
- Excel export работает или реализован базово.
- PDF export работает или реализован через упрощенный вариант.
- PNG export работает или реализован через упрощенный вариант.
- Печать через браузер работает.

## 12. Режим только просмотра

- В режиме просмотра нельзя редактировать.
- Нельзя drag-and-drop.
- Нельзя импортировать.
- Можно экспортировать.

## 13. Undo / Redo

- Undo работает для добавления, удаления, редактирования и drag-and-drop.
- Redo работает.
- Импорт не обязан пошагово отменяться.

# Implementation Notes

## 1. Главный принцип

Создать локальное offline-приложение без backend.

Приложение должно работать через `file://`, поэтому не использовать ES-модули, если они ломают запуск двойным кликом.

## 2. Рекомендуемые файлы

```text
index.html
styles.css
js/app.js
js/state.js
js/calculations.js
js/render.js
js/dragDrop.js
js/importExport.js
js/validation.js
js/undoRedo.js
libs/xlsx.min.js
libs/html2canvas.min.js
libs/jspdf.min.js
```

## 3. State-first подход

Сначала построить структуру состояния:

- project;
- settings;
- scenarios;
- offices;
- zones;
- teams;
- employees;
- allocations.

Все расчеты должны строиться из state, а не из DOM.

## 4. Порядок разработки

1. State model.
2. Scenario management.
3. Offices and zones.
4. Teams.
5. Employees.
6. Allocations.
7. Calculations.
8. Validation.
9. Dashboard rendering.
10. Manual allocation table.
11. Drag-and-drop.
12. JSON import/export.
13. Excel import/export.
14. PDF/PNG/print.
15. Undo/redo.
16. View-only mode.

## 5. Что нельзя упрощать

- Drag-and-drop.
- Расчеты.
- Сценарии.
- Размещения.
- Оцифрованное переполнение.

## 6. Что можно упростить

- PDF-дизайн.
- PNG-экспорт.
- Excel-импорт.
- Excel-экспорт.
- Onboarding.
- Пустые состояния.

## 7. Расчеты должны быть вынесены отдельно

Рекомендуемый файл:

```text
js/calculations.js
```

Функции:

- calculateOfficeCapacity(office)
- calculateOfficeOccupancy(project, scenario, officeId)
- calculateZoneOccupancy(project, scenario, zoneId)
- calculateOverflow(capacity, occupied)
- calculateScenarioKpis(scenario)

## 8. Валидация отдельно

Рекомендуемый файл:

```text
js/validation.js
```
